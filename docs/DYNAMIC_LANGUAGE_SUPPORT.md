# 动态验证语言能力与项目筛选约束

## 当前能力

| 语言 | HTTP 项目沙箱 | 函数级 Harness | 说明 |
| --- | --- | --- | --- |
| Python | 支持 | 支持 | 目标函数切片及框架路由增强。 |
| JavaScript / TypeScript | 支持 | 支持 | JavaScript 切片；TypeScript 按 JavaScript 运行时处理。 |
| PHP | 支持 | 支持 | 过程式函数切片。 |
| Ruby | 支持 | 支持 | 顶层方法切片。 |
| Go | 支持（需可执行 runtime plan） | **有限支持** | 仅顶层、字符串参数、可受控替换 `exec.Command` 的函数级切片；不支持任意 Go 依赖图、数据库、真实路由或 WebSocket 入口。 |

HTTP 沙箱是否可用，取决于项目运行时计划能否得到经过授权的本地 `base_url`；它不由 Harness 语言能力决定。

## 20 项目批次的硬性筛选规则

项目只有同时满足以下条件，才能标为“当前动态能力覆盖”：

1. 生产主语言位于上表支持范围；
2. 静态规则覆盖该项目使用的漏洞类型；
3. 对目标漏洞类型，至少存在可执行的 HTTP 计划或该语言支持的 Harness 路径；
4. 启动计划可由仓库证据证明，或用户明确提供授权的运行目标；
5. 对受限能力（当前 Go Harness），项目与验收结论必须明确标为“有限支持”，不能计作完整端到端动态覆盖。

不满足任一项的项目可以作为静态审计样本，但不得计入“已支持 Deep 动态验证”的交付数量。
